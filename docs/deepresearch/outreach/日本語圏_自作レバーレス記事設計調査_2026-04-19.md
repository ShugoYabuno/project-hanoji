## 結論

- 日本語圏で「自作レバーレス」文脈の継続発信者は、note を中心に複数確認でき、特に khide、OrangeCrush、ヤマカズ、dizzy236hs、kaniron などは、単発ではなく複数回の発信・派生記事・X 連動の痕跡が見られたため、一定以上の継続性があると判断できる。[1][2][3][4][5]
- 反応が取りやすい記事は、冒頭 100〜200 字で「何を作ったのか」よりも先に「なぜ困っていたのか」「誰向けの話か」「読むと何が分かるか」を約束していることが多く、結果として未経験者でも本文に入っていきやすい。[6][7][8][3]
- 「情報量が多く感じにくい記事」は、1 記事 1 テーマの徹底、見出しの細分化、各見出し内の論点の少なさ、箇条書きによる判断ポイントの可視化、画像を“読ませる前”に置く構成が共通していた。[9][8][10][11]
- 逆に内容は良いが重く見える記事は、製作背景・部品説明・配線・設計思想・完成後の感想が一記事に同居しやすく、専門用語も前提知識込みで流れるため、未経験者には「読む前から難しそう」に映りやすい。[12][13][14][1]
- エルゴノミクス寄りのレバーレスを未経験者に届けるには、「強い設計思想」より「手首が痛い」「小指がつらい」「普通のレバーレスだと置き方が合わない」といった身体感覚の困りごとから入るほうが刺さりやすい。[15][16][6]
- Project Hanoji では、販売臭を出すよりも「試作の透明性」「失敗の共有」「まだ決め切れていない点」を前面に出し、読者に“開発を見守る参加感”を持たせる構成が相性がよい。[2][8][4]
- v2 記事は、技術情報を全部載せるのではなく、冒頭では「何を解決したい試作なのか」「どこが前より良くなったか」「まだ未完成な点は何か」に絞り、詳細な配線・部材・実装は別記事化した方が、初見読者の離脱を抑えやすい。[7][10][3]

## 横断比較

### 冒頭の入り方

反応が良い記事の冒頭は、大きく 4 パターンに分かれた。第一に「困りごと起点」で、既存のアケコンやレバーレスに限界を感じた話から始める型である。第二に「初心者宣言型」で、自分は素人だが作れたと置く型で、読者の不安を下げやすい。第三に「衝動・感情型」で、結局作りたくなった、Version2 を作ってしまった、といった気持ちの勢いから入る型である。第四に「具体価値先出し型」で、薄型・スマホサイズ・はんだ不要のようなベネフィットを一行目から示す型で、タイトルと本文の約束が一致しやすい。[11][16][3][4][1][9][2][7]

Hanoji 向けには、エルゴノミクスの設計思想から入るより、「普通のレバーレスだと姿勢や手首がしっくり来なかった」という身体感覚の違和感から入る方が、未経験者にも理解されやすい。[16][6]

### 最初の 100〜200 字で何を約束しているか

強い記事は、最初の短い範囲で少なくとも次の 2 つを明示していた。ひとつは「この記事で何が分かるか」、もうひとつは「この記事は誰に向けたものか」である。たとえば「誰でも自作できる」「注意事項をまとめる」「必要なもの編」といった表現は、読者が自分に必要かどうかをすぐ判断できる。[8][10][3][7]

読みやすい記事ほど、最初の 100〜200 字で“全部は教えない”代わりに、“今回はここだけ分かれば十分”という範囲設定が明確だった。これは、情報量そのものを減らすだけでなく、読者の期待値を適切に狭めている点が重要である。[10][11]

### 1記事1テーマに絞れているか

最も読みやすいのは、「必要なもの」「注意事項」「Version2 の変更点」「設計入門」など、見出し以前に記事単位でテーマが 1 本化されている記事だった。逆に、背景、作り方、買った部品、配線、完成後レビュー、今後の構想までを同居させると、価値は高いが“重い記事”になりやすい。[14][3][12][1][2][8][10]

Project Hanoji は試作の透明性を武器にできるが、それを一記事に全部入れる必要はない。むしろ「今回の試作で分かったこと」だけに絞った方が、続きを読みたい気持ちを残しやすい。[4][2]

### 見出しの数と粒度

読みやすい記事は、全体のボリュームに対して見出し数がやや多めで、1 見出しの役割が狭い傾向があった。体感的には、2,000〜4,000 字級の記事で 4〜8 見出し程度に分かれている構成が多く、各見出しで扱う論点も 1〜2 個にとどめられている。[11][6][8][10]

一方で、重い記事は見出し数が少ないのに本文段落が長く、1 セクションの中で「背景→判断基準→具体物→感想」が連続して出てきやすい。これは情報量が多いというより、休憩ポイントが少ないことが重さの原因になっている。[13][1][14]

### 1見出しあたりの情報量

軽く読める記事では、1 見出しあたりの情報量は「判断材料 1 つ＋具体例 1 つ＋画像 1 枚」くらいで止められていることが多い。特に初心者向け記事では、見出しごとに問いがひとつだけ立っていると読み進めやすい。[9][7][8][10][11]

逆に、部材比較・配線・ケース構造・ファームウェア・使用感が同じ見出し内に混在すると、途中で“いま何の話か”が分かりにくくなる。[12][1][14]

### 箇条書きの使い方

反応が良い記事は、箇条書きを「網羅のため」ではなく「判断を速くするため」に使っていた。具体的には、必要なもの、注意点、良かった点、困った点など、読者が行動に直結させたい箇所だけをリスト化している。[3][8][10][11]

一方で、重い記事は長文の中に情報が埋まり、箇条書き化できる内容まで地の文で流していることがある。Hanoji では、「今回分かったこと 3 点」「まだ解決していないこと 2 点」のような短い箇条書きが効く。[1][2][8][12]

### 専門用語の説明の仕方

初心者に刺さる記事は、専門用語を定義から教えるのではなく、まず現象やメリットを先に書いてから、必要な場合だけ名称を添えていた。たとえば「左右分割」「薄型」「はんだ不要」などは、専門性より使用イメージを先に想像させる言葉として機能している。[6][7][11]

反対に、GPIO、Pico、Kailh、配線マトリクスなどの語が早い段階から連続すると、内容の質は高くても未経験者には“自分向けではない記事”に見えやすい。Hanoji で使う専門語は、身体感覚やレイアウトの文脈に紐づけて説明した方がよい。[15][16][12][1][9]

### 画像の置き方

読みやすい記事は、画像を「説明の補足」ではなく「次の段落を読む理由」として前倒し配置していることが多い。つまり、本文で長々と説明してから最後に完成写真を見せるのではなく、早い段階で完成形・問題箇所・改善前後を見せ、読者に具体像を持たせてから文字を読ませている。[8][9][6]

Hanoji のように形状や姿勢が価値になるプロジェクトでは、画像は特に重要である。真上写真だけではなく、「手を置いた状態」「普通のレバーレスとの差」「困っていた姿勢との比較」があると、未経験者でも価値が理解しやすい。[16][9]

### 締め方

反応を取りやすい記事の締め方は、完成宣言より「ここまでは分かった」「次はここを試す」「同じことで困っている人の参考になれば嬉しい」のような、開いた終わり方が多い。この形式は、販売臭を避けつつ、続きを待つ理由も作れる。[2][4][8]

特に試作ログ型では、完全な答えを出すより「未解決点を一つだけ残す」方が、読者の記憶に残りやすい。[2]

### コメントや反応を呼ぶ導線

露骨な「買ってください」導線ではなく、「同じ悩みがある人いますか」「この配置どう思いますか」「こういう押し方の人いますか」といった問いかけの方が、コミュニティ記事として自然だった。また、X で記事公開を継続告知している例もあり、note 単体ではなく SNS と組み合わせることで継続反応を作っている。[21][22][4][16]

Project Hanoji でも、「まだ最適解ではない」「試した人がいたら知りたい」という余白を残すと、試作の透明性と相性がよい。[8][2]

## 「情報量が多く感じにくい記事」の共通点

- タイトルで結論かテーマ範囲が分かる。「必要なもの編」「注意事項まとめ」「設計入門」など、読む前に記事の枠が見える。[10][3][8]
- 冒頭で対象読者を絞る。初心者向け、初めて作る人向け、困っている人向け、という明示がある。[7][11][8]
- 1 記事 1 テーマで、別トピックを大胆に切り出している。[3][10][2]
- 段落が短く、1 段落 2〜4 文程度で止められている記事が多い。[11][6][8]
- 各見出しの論点が狭く、読者が「ここだけ読めば一つ分かる」と感じやすい。[10][8]
- 画像が先、説明が後の場面があり、文字だけで理解させようとしていない。[9][6][8]
- 箇条書きが“記録”ではなく“判断支援”に使われている。[3][11][8]
- 専門用語を語る前に、身体感覚や使用シーンを置いている。[16][6][7]

## 「内容は良いが重い記事」の共通点

- 制作背景・設計思想・材料・配線・完成後レビューを一記事に詰め込みがちで、読者の目的が途中でぶれやすい。[14][12][1]
- 技術情報の密度が高いのに、休憩点となる見出しや箇条書きが少ない。[13][12][1]
- 書き手には当然の語彙でも、未経験者には用語の壁がある。[12][9]
- 製作記録としては価値が高いが、「この人の試行錯誤を追う前提」が必要で、初見には入りづらい。[1][2]
- 画像が完成記録として置かれていても、本文導線の一部としては機能していないケースがある。[14][1]
- 面白い内容でも、読者への約束が広すぎると、“長そう”に見えて読む前に離脱されやすい。[13][14]

## Project Hanoji 向け提案

### 優先順位つきで真似すべき書き方

1. **冒頭 120 字で「困りごと→今回の試作で変えた点→誰向けか」を言い切る**。特にエルゴノミクス寄りなら、姿勢・手首・指の負担から入るのが有効である。[7][16]
2. **1 記事 1 テーマを徹底する**。v2 全体紹介、配線詳細、部材選定、設計思想、使用感は分ける。[2][10][3]
3. **画像を冒頭近くに置き、「普通のレバーレスと何が違うか」を先に見せる**。文字で説得する前に、形の違いを理解させる。[16][9]
4. **“素人でも分かる説明”ではなく、“自作未経験でも想像できる説明”にする**。専門用語の定義より、身体感覚・使い方・置き方から説明する。[6][11][7]
5. **失敗や未解決点を1〜2個あえて残す**。完成度の高さより、透明性と共感が目的に合う。[4][8][2]
6. **箇条書きは「今回分かったこと」「まだ悩んでいること」に限定する**。部品一覧を長く書くより、判断材料として見せる。[8][3]
7. **締めで軽く問いを投げる**。同じ悩みの人、エルゴ寄りレバーレスに興味がある人へ、感想や経験の共有を促す。[4][16]
8. **販売臭は抑え、「試作ログを公開している理由」を書く**。これは単なる宣伝ではなく、試作の透明性と観察記録だと伝える。[4][2]

## Project Hanoji に適した記事テンプレート

### 失敗談型

**向いている場面**: 期待した改善がうまくいかなかった時、配置や角度やサイズで失敗した時。[2][8]

**構成**
1. 冒頭：何を良くしたくて試作したか、でもどこでズレたかを 3〜4 文で書く[8][4]
2. 失敗前提の写真：見た目は良さそうなのに使うとダメだった、が伝わる写真を置く[9][16]
3. 何がダメだったか：身体感覚ベースで 3 点以内に整理する[15][16]
4. なぜそうなったか：専門語は最小限で、配置・角度・支点など原因を短く説明する[6][16]
5. 次にどう直すか：次回試す変更を 1〜2 点だけ示す[2]
6. 締め：同じ悩みがある人に問いかける[4]

**冒頭例**
「手首を楽にしたくて Hanoji v2 を試したのですが、使ってみると想像と違う疲れ方が出ました。今回は『何がダメだったのか』だけを先に整理します。自作経験がない人でも、エルゴ寄りのレバーレスって何を難しくするのかが伝わるように書きます。」

### 試作ログ型

**向いている場面**: v1 から v2 への変化、試作の連続性を見せたい時。[3][2]

**構成**
1. 冒頭：前回のどこが不満だったか、今回は何を変えたかを先に示す[2]
2. 変更点サマリー：3 項目以内の箇条書き[3][8]
3. 変更ごとの短い見出し：1 見出し 1 変更点[10]
4. 触って分かったこと：想定通り／想定外を分けて書く[8][4]
5. 今後：次回の試作で見るポイントを一つだけ書く[2]

**冒頭例**
「Hanoji v2 では、前回しっくり来なかった“置き方”を優先して直しました。今回は完成報告ではなく、v1 からどこをどう変えたかだけに絞って書きます。エルゴノミクス寄りレバーレスに興味はあるけど、自作はまだ……という人にも伝わるよう、部材の細かい話は後回しにします。」

### エルゴノミクス解説型

**向いている場面**: プロジェクトの思想を伝えたいが、販売臭を出したくない時。[15][16]

**構成**
1. 冒頭：普通のレバーレスで困る身体感覚をひとつ挙げる[16]
2. なぜ起きるか：姿勢・腕の開き・手首の角度などを平易に説明する[15][6]
3. Hanoji で試している方向性：解決策を 2〜3 個だけ示す[16]
4. まだ分からないこと：断定しすぎず、個人差を認める[15]
5. 締め：実験として見てほしい、と開いて終える[4][2]

**冒頭例**
「レバーレスは速くて面白い一方で、長く触ると『なぜか置き場が決まらない』『手首だけ妙に落ち着かない』と感じることがあります。Hanoji は、その違和感を少しでも減らせないかを試しているレバーレスの試作です。今回は自作方法ではなく、なぜエルゴノミクス寄りにしたいのかをできるだけやさしく整理します。」

## Hanoji v2 記事の具体提案

### 削るべき情報

- 全パーツ一覧の細かい型番列挙[9][3]
- 配線や基板実装の詳細手順[12][14]
- ファームウェア設定の細部[12]
- 長い制作経緯の時系列説明[1][4]
- 比較のためだけの細かい寸法表を冒頭に置くこと[9]

これらは価値がないのではなく、初見向けの v2 記事では「読む理由」を作る前に情報負荷を上げてしまうため、別記事に逃がした方がよい。[10][3]

### 別記事に分離すべき情報

- 部材一覧と選定理由[3]
- v1 と v2 の寸法・配置差分の詳細[2]
- 実装方法、配線、ファームウェア[14][12]
- 完成後の長期使用レビュー[8]
- エルゴノミクス設計の深掘り解説[15][16]

この分離により、v2 記事自体は「興味を持たせる入口」、別記事は「知りたい人向けの深掘り」と役割を分けられる。[11][10]

### 冒頭に残すべき情報

- どんな違和感や課題が出発点だったか[16]
- v2 で一番変えたポイントは何か[2]
- 完成ではなく試作途中で公開する理由[4][2]
- この記事は誰向けか。エルゴ寄りに興味はあるが、自作経験はない人も含むと明示する[7][6]
- 完成写真または使用姿勢写真への導線[9][16]

### 見出し案

Hanoji v2 の主記事は、以下のような見出し粒度が適している。

1. **なぜ v2 を作ったのか** — 課題と狙いだけ[16][2]
2. **まず見てほしい形の違い** — 写真中心[9]
3. **今回いちばん効いた変更** — 1 点に絞る[10]
4. **逆に、まだ微妙だったところ** — 失敗や未解決を出す[8][2]
5. **自作未経験の人に伝えたいこと** — 難しさより面白さと入り口[11][7]
6. **次に試したいこと** — 続きを作る[2]

## Hanoji 用の短い設計原則

- 読者に渡すのは「全部の情報」ではなく「続きを読みたくなる解像度」で十分である。[11][10]
- 技術の凄さより、身体感覚の違いを言葉にする方が、エルゴノミクス系では強い。[15][16]
- 失敗や迷いを隠さない方が、販売臭が消えて信頼が増す。[4][2]
- 記録として正しい記事と、初見に刺さる記事は別物なので、Hanoji の主記事は後者を優先した方がよい。[10][3]
- 「今回はここまでしか話さない」をはっきり決めることが、結果として一番読みやすさに効く。[11][10][8]
